# AGENTS.md

本文件定义 TinyLLM-System 项目中 AI Agent、开发者和代码审查者必须遵守的协作规则。

---

## 1. 总体原则

1. 文档先于大规模实现。
2. 每个里程碑只解决一个主问题。
3. 不虚构 Benchmark、准确率、吞吐或显存结果。
4. 不因为“技术听起来高级”就提前加入功能。
5. 每个重要功能必须有 Smoke Test、失败路径和验收标准。
6. 所有训练结果必须能够追溯到数据、配置、代码和硬件。
7. 任何跨 GPU 策略都必须先验证正确性，再讨论性能。

---

## 2. Agent 禁止事项

AI Agent 不得：

- 一次性实现 Transformer、MoE、DDP、FSDP、ZeRO、KV Cache、API 和前端。
- 为了让 README 好看而填写未经运行的指标。
- 在没有显存探测或 Smoke Test 时宣称某配置一定可运行。
- 在 DDP 中声称多张 GPU 会自动合并显存。
- 未保存随机数、优化器和采样器状态就宣称“可精确恢复”。
- 忽略 V100 不支持 BF16 的差异。
- 默认使用只在 A100/H100 上稳定的算子或量化格式。
- 把 Notebook 作为正式训练入口。
- 将所有逻辑塞入单个 `train.py`。
- 在没有测试的情况下重构训练核心。

---

## 3. 开发顺序

每个功能按照以下顺序推进：

```text
设计文档
  → 最小接口
  → 单元测试
  → Smoke Test
  → 失败路径
  → 集成测试
  → 真实 Benchmark
  → 文档更新
```

任何功能在没有完成 Smoke Test 前，不得进入下一个复杂阶段。

---

## 4. 配置规则

- 所有正式实验必须从 YAML 配置启动。
- CLI 参数只允许覆盖少量运行时字段。
- 训练开始后保存完整配置快照。
- 配置必须经过 Schema 校验。
- 配置中必须显式记录：
  - 模型。
  - 数据版本。
  - Tokenizer。
  - 精度。
  - 并行策略。
  - GPU 数量。
  - Batch Size。
  - 序列长度。
  - Checkpoint 策略。
  - 评测套件。

---

## 5. 数据规则

- 原始数据不得直接被训练器读取。
- 所有数据必须经过统一预处理并注册版本。
- 训练、验证和测试必须在去重后划分。
- Manifest 必须记录输入哈希、处理配置和输出哈希。
- 被过滤样本必须保留数量和原因统计。
- 数据格式变化必须提升 Schema 版本。
- 不允许用测试集调参。

---

## 6. 训练规则

### 6.1 正确性优先

在扩展到多卡之前，必须通过：

- CPU Smoke Test。
- 单卡前向。
- 单卡反向。
- Loss 下降。
- Checkpoint 保存与加载。
- 中断恢复。

### 6.2 分布式训练

DDP、FSDP2 和 ZeRO-3 必须分别验证：

- 参数初始化一致。
- Global Batch 计算正确。
- Loss 聚合正确。
- Rank 0 日志不重复。
- 分布式 Checkpoint 可读取。
- 异常退出后可恢复。
- 不同 World Size 的结果差异在合理范围内。

### 6.3 精度策略

- RTX 3090 默认 BF16。
- V100 默认 FP16 + GradScaler。
- 不允许在 V100 配置中启用 BF16。
- TF32 仅在支持的 GPU 上启用。
- 数值异常必须记录 Loss Scale、Gradient Norm 和首个异常 Step。

---

## 7. Checkpoint 规则

Checkpoint 至少包含：

- 模型参数。
- 优化器状态。
- 学习率调度器。
- GradScaler。
- 当前 Epoch 和 Step。
- Python、NumPy、PyTorch、CUDA RNG 状态。
- 数据采样器状态。
- 数据版本。
- 训练配置快照。
- Git Commit。
- 环境信息。
- Checkpoint 完整性元数据。

恢复时必须先校验：

- 文件完整性。
- 模型结构兼容性。
- 数据版本。
- World Size 约束。
- 精度策略。
- 配置差异。

---

## 8. 评测规则

- 评测必须与训练进程解耦。
- 评测集必须版本化。
- 任何人工判断指标必须保存判定依据。
- 模型比较必须同时报告提升与回退。
- Promotion Gate 的阈值必须来自配置。
- 未通过门禁的模型不得标记为 candidate 或 production。
- Benchmark 结果必须记录：
  - 模型版本。
  - 硬件。
  - 软件版本。
  - 并发。
  - 输入输出长度。
  - Batch 设置。
  - 运行次数。

---

## 9. 代码质量

- 使用类型标注。
- 公共接口必须有 Docstring。
- 核心逻辑不得静默吞掉异常。
- 对外错误应有明确错误码和上下文。
- 日志中不得泄露密钥和完整隐私数据。
- 单元测试与集成测试分开。
- GPU 测试必须带 Marker，避免 CI 默认执行。
- 任何性能优化不得绕过正确性测试。

---

## 10. 里程碑完成条件

一个里程碑只有同时满足以下条件才算完成：

1. 代码合并。
2. 测试通过。
3. Smoke Test 成功。
4. 失败路径验证。
5. 文档更新。
6. 真实结果记录。
7. 下一阶段依赖明确。

“代码写完”不等于“里程碑完成”。

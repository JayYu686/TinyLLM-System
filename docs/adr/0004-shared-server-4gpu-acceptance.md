# ADR-0004：共享服务器采用四卡正式验收上限

## 状态

Accepted；在 M3 起生效，并取代 ADR-0002 中“8 卡是正式验收必需项”的部分。

## 背景

主服务器物理上有 10 张 RTX 3090，但长期由多个用户共享。实际观测显示，GPU 0–4
经常运行其他用户的长期任务，项目不能停止、抢占或绕过 Preflight 使用这些进程。
因此，等待 8 张卡同时空闲没有可规划的完成时间，并会让 DDP、FSDP2 和后续求职版本
被资源排队而非工程正确性阻塞。

GPU 数量是实验条件，不是项目专业性的替代指标。专业性由可复现协议、正确的并行语义、
失败恢复、完整血缘和对结果边界的诚实说明共同证明。

## 决策

1. M3 的正式 DDP 扩展矩阵固定为 1/2/4 卡 Strong Scaling 与 Weak Scaling；每个单元
   仍执行预热 20 Step、测量 100 Step 和 3 次独立重复。
2. 正式矩阵使用嵌套且通过 Preflight 的 GPU 集合：单卡 GPU 8、双卡 GPU 8–9、四卡
   GPU 6–9。若这些卡不空闲，实验等待或改期，不抢占未知进程。
3. 8 卡 DDP 和 GPU 4–7 跨 NUMA 对照改为非阻塞增强证据。只有两种 NUMA 分组都完成
   同协议实测时，才允许给出受控 NUMA 性能比较。
4. 已完成的 GPU 6–9 同 NUMA 四卡结果作为补充证据保留；在缺少跨 NUMA 配对结果时，
   必须标记为 `partial`，不能推导跨 NUMA影响。
5. M4 的正式 FSDP2 目标改为四卡。必须先用真实显存 Probe 验证 Qwen3-8B 配置；如果
   四卡仍无法满足目标，不得宣称可运行，需通过新 ADR 调整模型、序列长度或验收目标。
6. 所有公开文档、报告和简历只描述实际完成的最大 World Size，不把四卡结果外推为
   八卡或十卡结论。

## 保留 ADR-0002 的部分

ADR-0002 关于“不默认使用全部 10 卡”、显式记录拓扑、保留评测/开发资源和例外实验需
单独说明的原则继续有效。其固定 GPU 0–7 八卡组不再是核心发布门禁。

## 后果

- M3 能在共享资源上按可预测节奏完成，并形成完整的 1/2/4 卡扩展曲线。
- Strong Scaling 的 Global Batch 8 仍可分别使用 8/4/2 次 Gradient Accumulation，
  不需要重跑已有 1/2/4 卡结果。
- 项目不再声称验证了 8 卡扩展效率或受控跨 NUMA 性能差异。
- 未来若获得可重复的独占 8 卡窗口，可以用同一 Harness 追加完整 Strong/Weak 证据；
  可选证据缺一项时不得标记为完整。

## 重新评估条件

只有在能够预先协调稳定的 8 卡独占窗口，且不会影响其他用户任务时，才重新讨论将 8 卡
恢复为发布门禁。一次偶然空闲不足以改变本决策。
